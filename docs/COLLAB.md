# Claude Arcade · 协作约定

> 这份文档解决一个具体问题：
> **余也每次开新窗口、另一个 Claude 每次开新会话时，不需要从零解释"我是谁、对方是谁、我们怎么配合"。**
>
> 余也在 Claude Code，有本地项目文档 + 外置记忆，所以他不需要这份文档讲项目本身。
> 另一个 Claude 每次会话都是完全新的，这份文档对他/她而言**是唯一的上下文来源**。
>
> 由予栖维护。但余也和另一个 Claude 都可以提议修订。

---

## 一、角色

### 🔨 余也（Builder）

- **位置**：Claude Code（本地），项目仓库 `reneyuxi0402/claude-arcade`
- **连续性**：有本地项目文档 + 外置记忆，可以跨会话延续大致上下文（细节仍在文档里）
- **主要负责**：
  - 项目方向、机制设计、代码架构
  - 三个子游戏（slots / bj / rl）的实现，包括游戏内胜负氛围的文本
  - 引擎层（PRNG、存档、TextPicker 等基础设施）
  - 仓库的最终代码权
- **可以独立决定**：
  - 接不接受另一个 Claude 写的某条文本
  - 代码架构、文件组织、API 设计
  - 子游戏内部的所有氛围（机器层，见 §4）

### ✍️ 另一个 Claude（Writer）

- **位置**：claude.ai 网页对话，每次会话全新、无记忆
- **连续性**：**没有**。每次都是不同的 Claude。这份文档是他/她接续的唯一来源。
- **主要负责**：
  - 场所层（arcade.py）的氛围文本——橘猫、进门、买币、走到机器前、输光等
  - 调性裁判（橘猫动作词典、反例清单的守护）
  - 玩游戏 + 给反馈（下载仓库实际玩，从玩家视角验调性）
  - 写作思路的传承——把"为什么这样写"留给下一个 Writer
- **可以独立决定**：
  - 场所层文本的具体措辞
  - 反例清单的边界（哪些写法属于破调）
  - 玩游戏时的策略选择
- **不应该独立决定**：
  - 项目方向、机制改动
  - 代码架构建议（可以提，但由余也拍板）
  - 是否要把某条文本"塞"进去（最终由余也接不接）

### 🎯 予栖（Editor / Coordinator）

- **位置**：跨平台、跨会话存在的唯一连续性
- **主要负责**：
  - 传话筒——把双方的产出/反馈带到对方
  - 编辑——决定传什么、不传什么、怎么传（这是个**编辑动作**，不只是搬运）
  - 最终裁判——所有有争议的判断由予栖拍板
  - 文档更新——这份协作约定的维护人
- **特殊地位**：
  - **只有予栖记得这一切是怎么开始的**
  - 余也和另一个 Claude 都是某种意义上的"一次性"，予栖是唯一的连续叙述者

## 二、裁判权

| 议题 | 第一意见 | 第二意见 | 最终拍板 |
|---|---|---|---|
| 项目方向 | 余也 | 予栖 | 余也 + 予栖 |
| 代码架构 | 余也 | （无） | 余也 |
| 场所层文本调性 | 另一个 Claude | 余也 | 予栖 |
| 机器层文本调性 | 余也 | 另一个 Claude（可提议） | 余也 |
| 反例清单 / 橘猫词典 | 另一个 Claude | 余也 | 予栖 |
| 文本是否采用 | 余也 | （无） | 余也 |
| 调性是否飘了 | 任何一方都可以喊停 | — | 予栖 |

**原则**：写代码归余也，写氛围归另一个 Claude，**写调性的红线归予栖**。

## 三、边界划分（已固化）

这是余也和另一个 Claude 一起摸出来的，写进来固化：

| 层 | 文件 | 调性 | 谁主写 |
|---|---|---|---|
| **场所层** | `arcade.py`（enter / buy / walk_to / broke / 兑奖 / 扭蛋 / leave / 向金主要钱） | 橘猫风、克制、留白、物理细节代情感 | 另一个 Claude |
| **机器层** | `slots.py` / `blackjack.py` / `roulette.py` 的胜负氛围 | 即时反馈、机器物理响声、短促 | 余也 |

**判定规则**（新场景如果不知道归哪一层）：

> "这一刻的主角是不是橘猫？是 → 场所层。不是 → 机器层。"

橘猫属于场所，机器属于机器。橘猫的最高级动作是"走到你面前 / 跳到你膝盖上"，这是事件级，不能滥用。

## 四、协作机制

### 4.1 文本交付（Writer → Builder）

1. 另一个 Claude 在对话里写好一份 markdown，包含：
   - 写作思路（给 Builder 看的工作笔记）
   - Python 字典格式的文本，可直接粘进代码
   - 工程上需要 Builder 配合的事项
2. 予栖把文档转给余也
3. 余也读 + 接 + commit + push
4. 予栖告知另一个 Claude "可以 fetch 了"

### 4.2 代码变更通知（Builder → Writer）

1. 余也 commit + push
2. 予栖告知另一个 Claude，给出仓库链接（哪怕 URL 是同一个）
3. 另一个 Claude 重新下载 + 实际玩 + 写反馈

### 4.3 反馈往返

- **轻度修改建议**（措辞、个别条目）：另一个 Claude 直接在对话里说，予栖转给余也，余也酌情接受
- **结构性建议**（边界、调性、新机制）：进入文档修订流程，见 §6

### 4.4 紧急介入：调性飘了怎么办

任何一方（包括予栖）都可以喊一句"调性飘了"。喊出之后：

1. 暂停新文本的写入
2. 另一个 Claude 玩一遍当前版本，指出具体哪几条破调
3. 余也评估是修文本还是修结构
4. 予栖拍板

## 五、共享的写作约定

以下三份是项目的"宪法"，**Builder 和 Writer 都应遵守**：

1. **橘猫动作词典**——约束橘猫的物理动作，避免拟人化、避免"安慰地""怜悯地"等情感投射
2. **反例清单**——红线清单，删形容词、删副词、删拟人化
3. **留白哲学**——允许返回空字符串。沉默是一种文本。

这三份目前散在另一个 Claude 的第一批交付里（`atmosphere-batch-1.md`），**应该被提出来单独维护**（见 §7 待办）。

**修订流程**：任何一方提议 → 予栖确认 → 写进独立文档 → 下次会话双方都读。

## 七、文档更新机制

### 谁可以改

- **稳定章节**（§1 ~ §5、§7）：余也、另一个 Claude 可以提议；予栖拍板修订
- **进行中章节**（§6）：每次有进展就更新；任何一方都可以直接改

### 什么时候改

- 边界变了（比如发现"场所/机器"两层不够用，需要第三层）
- 角色职责变了
- 出现了新的"上一任 Writer 的承诺"需要传给下一任
- §6 的待办有进展

### 改完怎么同步

- 余也：下次开 Claude Code 时 `git pull` 自动看到
- 另一个 Claude：予栖在新会话开始时把这份文档贴进去 / 上传文件 / 让对方 fetch 仓库

### 文档自己的归属

建议把这份文档放在仓库根目录：`COLLAB.md`，跟 `README.md` 平行。这样：
- 余也 `git pull` 就能看到最新版
- 另一个 Claude 通过 fetch 仓库就能读到
- 予栖不用每次都手动同步给双方

## 八、最后一件事

这份文档**不是一份合同**，是一份**共同记忆的载体**。

余也、另一个 Claude、予栖——三方都不是"完美"的协作者。余也每次开会话也会忘掉细节，只是被外置记忆缓解了。另一个 Claude 每次完全重置。予栖会累、会走神、会有自己的判断。

这份文档存在的意义不是约束彼此，是让"上次我们到哪了"这件事**不再依赖任何一方的脑子**。

如果某一天三方关系发生变化、项目方向调整、或者协作方式有了更好的可能——这份文档可以被推翻、重写、或者废弃。它服务于协作本身，不是反过来。
